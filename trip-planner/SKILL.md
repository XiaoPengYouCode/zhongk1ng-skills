# 行程规划

这个 skill 处理私人出行。

核心目标很直接：先站在现成订票 skill / MCP / 浏览器自动化项目的肩膀上抓到实时选项，再把请假成本折成钱，最后只给一个推荐方案和少量备选。

明确约束：

- 不使用 `lark-*` 技能。
- 优先走公开 web 信息和公开可复用项目。
- 面对实时票价、余票、航班信息时，必须附来源链接和查询时间。
- 遇到法定节假日、调休、补班日期时，先核对官方放假安排，再计算请假成本。
- 火车方案必须经过 `12306` 官方实时余票接口或等价 `12306-mcp` 校验，单纯时刻表页不能作为推荐依据。
- 机票方案必须来自真实可订链路：优先 flight MCP / 订票 API，其次是带具体航班卡片和 `预订` 入口的实时订票页。
- 机票成本必须按总支付成本算：票面价 + 机建 + 燃油 + 必要行李费。节假日燃油或税费上浮时，必须单独计入。

## 先找现成能力

开始规划前，先读 [references/external-stack.md](references/external-stack.md)。

优先检查当前环境里有没有这些能力可直接复用：

1. `12306-mcp`
2. `12306 Train Assistant`
3. `Tuniu Flight`
4. 通用 browser automation

使用顺序：

- 火车票优先 `12306` 系能力。
- 机票优先 `Tuniu Flight` 或等价 flight MCP。
- 都没有时，退到浏览器自动化直接查官方站或主流订票站。

默认把上海也当成长三角核心换乘枢纽处理：

- 机场：`SHA`、`PVG`
- 车站：`上海`、`上海虹桥`

火车票额外规则：

- 先查 `12306` 官方实时余票，再看时刻表细节。
- 查询结果里必须同时确认三件事：这天有这班车、当前可预订、目标坐席不是 `无票`。
- 如果只有时刻表，没有实时余票，不给"最优方案"结论，只能给"候选方案"。

机票额外规则：

- 优先使用 flight MCP 或可直接查询的订票 API。
- 如果当前环境没有可用 API key，就退到实时订票页，但页面里必须同时看到：日期、起降时间、价格、航司/航班、`预订` 或等价下单入口。
- 只有"历史均价""最近 90 天平均价""泛化推荐页"时，不给最终推荐。
- 如果页面价格看起来是裸票价，不含税费，就必须继续进入下单前页面，拿到税费拆分后再比较。
- 如果无法拿到税费拆分，只能把该机票方案标成"待核税费"，不能进最终推荐位。
- `baggageInfo` 为空时，默认按"不含免费托运额"处理，不能把它算成含 20kg。
- 用户没明确说带托运行李时，默认同时给两列成本：`随身行李版` 和 `含 20kg 托运版`；最终推荐时把差异直接写出来。

## 默认输入

如果用户没主动给全，先从上下文里补齐这些字段：

- 出发地、目的地
- 去程日期或时间窗
- 最晚到达时间
- 返程日期或时间窗
- 最早可出发时间
- 是否接受红眼航班、夜间到达、换乘、跨城机场
- 行李、舱位、座位偏好
- 日薪
- 请假最小粒度

默认值：

- 日薪 `1000` 元
- 请假粒度 `0.5` 天
- 半天请假成本 `500` 元
- 如果用户说"`4.30` 或 `5.1` 到"，默认把它理解成首选在 `2026-05-01 23:59` 前到；如果用户补充"`5.2` 也能接受"，就把截止时间扩到 `2026-05-02 23:59`

## 🔴 第一步：路径枢纽城市筛查（不可跳过）

**这一整节不是可选建议，是强制指令。** 在没有完成枢纽筛查之前，禁止开始任何搜索、任何查询、任何价格比较。

直达票搜完就停 = 方案不完整。理由：一个直飞票价 2000 块，但先飞到中途枢纽再坐高铁到目的地可能只要 900 块。这种方案不会出现在直达搜索里。经济到达的核心逻辑：先凑到一个大枢纽，再通过末端短途铁路/公路接驳。

### 筛查方法

查阅 [references/hub-cities.md](references/hub-cities.md) 获取全国按区域整理的交通枢纽数据（站码 + 机场码），然后：

1. **画线**：确定起点到终点的铁路干线走向（参考 [references/hub-cities.md](references/hub-cities.md) 中的主要铁路干线），列出途经的所有省份
2. **沿线取点**：途经省份至少取所有省会、主要铁路枢纽站、主要民航机场
3. **两侧扩展**：向沿线两侧各扩展一层相邻省份，至少取省会
4. **终点圈**：终点 1-2 小时铁路圈内所有城市站，全部纳入
5. **起点圈**：起点出发普速/动车 4-6 小时内可到达的所有城市
6. **补充**：用户额外点名的城市、跨省大型中转站（如郑州、武汉、长沙）

**原则：经济到达优先。** 宁可搜 30 个枢纽再过滤，不要漏掉低价中转点。

### 示例：南京 → 深圳

途经省份：江苏 → 安徽/浙江 → 江西 → 福建 → 广东，走京沪线南下转杭深沿海线

| 区域 | 铁路枢纽 | 航空枢纽 |
|------|---------|---------|
| 江苏 | 南京、镇江、常州、无锡、苏州、徐州 | NKG, CZX, WUX |
| 上海 | 上海、上海虹桥 | SHA, PVG |
| 浙江 | 杭州、宁波、温州、金华、嘉兴 | HGH, NGB, WNZ |
| 安徽 | 合肥、芜湖、蚌埠、黄山、马鞍山、宣城 | HFE, TXN |
| 江西 | 南昌、九江、赣州、上饶、吉安 | KHN, KOW |
| 福建 | 福州、厦门、泉州、漳州 | FOC, XMN, JJN |
| 广东(珠三角) | 广州、深圳、东莞、佛山、惠州、珠海 | CAN, SZX, HUZ |

对每个枢纽至少查 3 样：
1. 从南京能不能经济到达（普速优先，动车其次）
2. 从该枢纽能不能到深圳
3. 该枢纽有没有机场，有没有便宜航班飞深圳（注意机建 + 燃油差异）

### 智能换乘默认城市池

对用户没明确说但允许换乘的情况，按上述筛查逻辑从 [references/hub-cities.md](references/hub-cities.md) 动态生成城市池。不再使用单一硬编码列表。

扩点原则至少要覆盖：
- 终点的 1-2 小时铁路圈内所有城市站
- 起点出发的低成本普速/动车 4-6 小时可及城市圈
- 民航票价便宜的中途枢纽机场

没有独立民航机场的城市，折算成最近可用机场 + 铁路接驳。例：湖州 → HGH 或 SHA。

只有一种情况可以停下来问：继续推断会让推荐方向反过来，比如城市搞错、日期搞错、用户到底是单程还是往返不清楚。

## 请假成本模型

这个 skill 默认把"请假的机会成本"直接并进总成本。

如果用户没给工作时段，使用这组默认工作块：

- 上午 `09:00-13:00`
- 下午 `14:00-18:00`

计算规则：

1. 统计去程和返程各自从出门到可正常开始办事的占用时段。
2. 只要碰到一个工作半天块，就记 `0.5` 天请假。
3. 请假成本 = `占用半天数 * 500`。
4. 总经济成本 = `票价 + 必要接驳成本 + 请假成本 + 必要住宿成本`。

如果用户明确说某天本来就休息、远程办公、或者时间不算请假，直接按用户约束覆盖默认模型。

日期语义默认这样解释：

- 用户说"`5/5 返程`"，默认按 `5/5` 这天出发理解，哪怕落地跨到 `5/6` 凌晨，也要单独标出来。
- 用户说"`5/6 10:30 前到深圳`"，默认按"飞机在深圳落地时间不晚于 `2026-05-06 10:30`"作为硬约束。

如果方案不是从用户所在城市直接发车或起飞，例如"深圳出发，但去广州白云站坐车"，前置跨城短接也必须计入：

- 金钱成本
- 占用时间
- 因此前置短接产生的额外请假成本

### 节假日校正

如果出发或到达日期落在法定节假日内，默认该时段请假成本记为 `0`。

需要额外核对两件事：

- 节前最后一个工作日是不是还能晚间出发而不请假
- 调休补班日是不是反而要重新计入工作块

节假日信息优先使用国务院或地方政府转发的官方通知。

## 方案评估

对每个候选方案都统一整理成这几个字段：

- 交通方式
- 订票渠道
- 来源链接
- 查询时间
- 官方可售状态
- 目标坐席数量
- 出发/到达时间
- 门到门总耗时
- 票价
- 机建与燃油
- 出发侧前置短接
- 接驳或托运行李等硬成本
- 请假半天数
- 请假成本
- 总经济成本
- 换乘次数
- 明显风险点

排序规则：

1. 先排除不满足到达时间、预算、偏好的方案。
2. 再排除没有通过官方实时可售校验的方案。
3. 再按 `总经济成本` 从低到高排。
4. 同成本时，优先选更早可用到达时间。
5. 还相近时，优先直达、少换乘、少红眼、少跨城折腾。

## 主动搜索的绕行方案

**依赖第一步筛查结果。没完成第一步就不能进入此节。**

绕行方案基于第一步筛查出的枢纽城市池生成。必须对池中每个城市做完整搜索，不能拿到第一个有效结果就停。

至少搜索这四类绕行组合：

1. **夜间动卧 + 短途接驳**：过夜列车到中途大枢纽，再接 1 小时左右短途高铁
2. **全程普速或普速 + 普速**：从出发点周边普速站出发（如深圳周边可扩展到广州、东莞、惠州），沿中途枢纽逐段换乘
3. **终点周边落点扩展**：终点 1-2 小时铁路圈内所有车站，再接公路/公交
4. **飞 + 铁**：先飞到枢纽城市机场，再接 12306 官方可售短途列车

搜索范围从第一步筛选的枢纽城市池中取。如果第一步覆盖面偏窄，至少补上：
- 出发城市周边 1 小时普速可及的所有车站
- 终点同省及相邻省的省会机场
- 终点 1-2 小时铁路圈内所有车站

默认缓冲：

- 飞机落地到下一段铁路发车，至少预留 `90` 分钟
- 同城机场到火车站，优先选地面交通简单的主站
- 如果下一段只剩无座或候补，默认降级优先级

只有当绕行方案的总经济成本更低，且换乘风险没有明显失控时，才把它提到推荐位。

### 同城换乘缓冲

不要把所有城市都粗暴按 `90` 分钟处理。

至少对这些机场-车站对做单独缓冲与接驳成本估算：

- `上海浦东 -> 上海站`：默认 `120` 分钟，接驳成本按最省钱公共交通估 `7-10` 元
- `上海浦东 -> 上海虹桥`：默认 `150` 分钟，接驳成本按最省钱公共交通估 `8-10` 元
- `上海虹桥机场 -> 上海虹桥站`：默认 `45` 分钟，接驳成本可记 `0`
- `上海虹桥机场 -> 上海站`：默认 `75` 分钟，接驳成本估 `5` 元
- `南京禄口 -> 南京南`、`杭州萧山 -> 杭州东`、`无锡硕放 -> 无锡/无锡东`、`宁波栎社 -> 宁波站`：没有更精细数据时，先按 `90` 分钟和公共交通低成本估值处理

如果落地机场和后续车站的真实地理关系说明 `90` 分钟根本接不上，就直接判死，不要为了便宜硬凑。

## 工作流

### 闸门：第一步不可跳过

**在执行任何搜索、任何查询之前，必须先完成枢纽筛查。这是硬性闸门，不是可选建议。**

先回答自己这三个问题，缺一个就不能开始搜票：
1. 起点和终点之间经过哪些省份？走哪条铁路干线？
2. 沿线 + 两侧扩展 + 终点圈 一共列出了哪些城市？
3. 这些城市里哪些有机场？哪些是铁路枢纽？

三个问题都答完了，才算通过闸门，才可以进入第 2 步。

1. **🔴 路径枢纽城市筛查（闸门）**：先读 [references/hub-cities.md](references/hub-cities.md)，按"第一步：路径枢纽城市筛查"的方法，把起点到终点之间所有可能的中转枢纽城市找出来。这一步跳过 = 方案作废。必须输出筛查后的枢纽城市清单才能进入下一步。
2. 检查 [references/external-stack.md](references/external-stack.md) 里的公开积木能不能直接用。
3. 火车先走 `12306` 官方实时余票链路：`leftTicket/init -> leftTicket/query/queryG`，或走等价 `12306-mcp`。对第一步筛选出的每个枢纽城市，查起点→枢纽、枢纽→终点的两段车次。
4. 机票查实时可订选项，优先 API / MCP，拿不到时退到真实订票页。对第一步筛选出的每个枢纽城市，必须分别查从该枢纽飞终点的航班（不要在查到第一个直达票后就停手）。
5. 机票候选继续走 `searchLowestPriceFlight -> multiCabinDetails`。如果 `multiCabinDetails` 返回 `cabinPriceId`，就把它视为"已进入预订详情页"的硬校验通过。
6. 机票若 API 未直接给税费拆分，则额外按官方规则补齐：国内航班 `民航发展基金 50 元`，燃油附加费按当前官方标准判断 `<=800km` 还是 `>800km`。
7. 对每个机场落点，继续查落地后 `12306` 官方可售短接车。
8. 如果首段不是从用户所在城市出发，补上出发侧前置短接。
9. 把每个候选方案折成统一成本模型。
10. 额外单独给出"全程普速最便宜方案"，因为它经常不是总榜第一，但会是最低现金支出方案。
11. 给出一个明确推荐方案，再给 1 到 2 个备选。
12. 输出时直接写结论，别把用户丢进一大堆没排序的票池里。

如果当前会话里已经把官方结果缓存到了本地，先运行：

- `node trip-planner/scripts/recompute_from_cache.js`

这个脚本会统一做三件事：

- 解析 `12306` 原始余票串，拆成明确坐席数量和票价
- 按真实日期时间重算可接上的中转组合
- 把航班税费、请假成本、短接成本折进同一张比较表

## 下单前核验

最终给推荐前，必须做这一步。

### 机票

至少确认这些字段：

- 航司和航班号
- 起降机场与时间
- 票面价
- 机建 / 民航发展基金
- 燃油
- 行李规则
- `cabinPriceId` 或等价预订详情标识
- 页面存在 `预订`、`去下单` 或等价按钮，或者 API 已经返回可继续下单所需的预订详情主键

任意一项拿不到，就把该机票方案降级为候选，不进最终推荐。

### 火车

至少确认这些字段：

- 车次
- 这天确实开行
- 页面处于可预订状态
- 用户目标坐席的明确余票数量

余票字段不要只保留原始串，要格式化成能直接读的结构，例如：

- `二等座: 6`
- `一等座: 无票`
- `无座: 14`
- `候补: 可候补`

如果只有 `*`、`暂售至`、`起售时间未到` 这一类状态，也不能当成当前可下单。

## 输出格式

默认输出包含三块：

- 推荐方案：一句话结论，说明为什么它最值。
- 备选方案：只保留 1 到 2 个真正有竞争力的备选。
- 计算依据：把票价、请假成本、总经济成本、官方可售状态和关键权衡列清楚。

如果用户要继续到下单：

- 优先进入现成订票 skill / MCP 的下单流程。
- 否则使用浏览器自动化进入官方或可信站点。
- 没拿到确认页或订单号之前，不要声称"已经订好"。
